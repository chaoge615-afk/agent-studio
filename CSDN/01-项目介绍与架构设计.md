## 前言

说实话，搞 AI Agent 开发到一定阶段，你会发现一个很尴尬的问题——Agent 越做越聪明，但管理它的方式却越来越原始。你有一堆 MCP 工具、三层记忆系统、安全护栏策略，结果呢？全靠命令行和配置文件来操控。每次想看一眼审计日志，得去翻 SQLite 文件；想调整工作流节点，得改 Python 代码重新部署。这不行，得有个像样的管理界面。

这就是我造 **Agent Studio** 的初衷。它不是又一个 Agent 框架，也不是什么花哨的 Demo 项目——它是一个正经的**可视化管理控制台**，让你能用浏览器管理整个 Agent 平台的所有方面：看仪表盘、编排工作流、查看记忆、审查日志、配置安全策略，全都在一个 React 界面里搞定。

这篇文章是系列的第一篇，我会从宏观视角介绍这个项目存在的意义、它在整个系统中的定位、技术架构是怎么设计的。如果你也想给自己的 Agent 系统加一个管理面板，或者对 BFF（Backend For Frontend）架构模式感兴趣，这篇应该能给你一些启发。

## 一、为什么需要 Agent 可视化管理平台？

在聊架构之前，先说说痛点。我的 Agent 平台（agent-platform）是基于 LangGraph 构建的，具备完整的问答能力：意图分类、SQL 查询、RAG 检索、结果融合、反思记忆，一应俱全。但用起来有几个很实际的问题：

**1. 黑盒感太强。** 用户提了一个问题，Agent 内部走了什么路由？用了 SQL 还是 RAG？记忆系统有没有命中？这些信息都藏在日志里，没有一个直观的视图来看。

**2. 工作流调整成本高。** LangGraph 的节点和边是写在 Python 代码里的。想加一个节点？改代码、测试、重新部署。对于非开发人员来说，这基本等于不可能。

**3. 运维信息分散。** MCP 工具是否在线？记忆系统存了多少条数据？安全护栏拦截了多少次？这些运维指标分散在不同的服务和接口里，没有统一的入口。

所以我的解决方案很明确：**建一个独立的管理控制台，把所有运维和管理操作集中到一个 Web 界面里。** 不是做一个聊天前端（那个 agent-platform 自己就有），而是做一个面向开发者和运维人员的**管理面板**。

## 二、三件套演进：从数据采集到智能中枢到管理控制台

Agent Studio 不是一个孤立的项目，它是我整个 Bilibili 智能知识库系统的第三个组件。理解这三个项目的关系，才能理解 Agent Studio 的设计为什么是这个样子。

| 项目 | 定位 | 核心技术 | 端口 |
|------|------|----------|------|
| **content-analysis-system** | 数据采集与处理引擎 | Docker Compose + DuckDB + ChromaDB | 80, 8000, 8010, 8088, 8090, 9001-9003 |
| **agent-platform** | AI Agent 智能中枢 | LangGraph + FastAPI + MCP + ChromaDB | 8001 |
| **agent-studio** | 可视化管理控制台 | React + TypeScript + FastAPI | 5173, 8002 |

它们的关系可以用一条清晰的数据链路来描述：

```
视频采集 → ASR 转录 → 内容精炼 → 知识入库 → 智能问答 → 可视化管理
   ↓            ↓           ↓           ↓           ↓           ↓
Bilibili    Whisper/    LLM 精炼    DuckDB +    LangGraph    React
API 爬取    云端 ASR    + 分类标注   ChromaDB    + MCP + RAG   + ReactFlow
```

简单来说：

- **content-analysis-system** 负责"脏活累活"——从 B 站爬视频、转录语音、精炼内容、存到数据库里。它是数据的生产者。
- **agent-platform** 负责"脑力活"——接收用户问题，通过 LangGraph 工作流进行意图分类、查询路由、结果融合，最终给出答案。它是智能的中枢。
- **agent-studio** 负责"面子工程"——提供可视化的管理界面，让开发者能监控、配置、调试上面两个系统。它是运维的入口。

这三个项目独立部署、独立开发，通过 HTTP API 松耦合地连接在一起。这种设计让我可以单独升级任何一个组件而不影响其他两个。

## 三、系统架构总览

下面是整个系统的拓扑图。Agent Studio 位于最上层，通过 BFF 后端与 agent-platform 通信：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          用户访问层                                          │
│                                                                             │
│   Agent Studio 前端 (:5173)          Content Frontend (:80)                 │
│   React + ReactFlow + Recharts       React + Nginx                          │
│   管理控制台                          数据查询界面                            │
└──────────────┬──────────────────────────────┬───────────────────────────────┘
               │ /api proxy                   │ /api, /query proxy
               ▼                              ▼
┌──────────────────────────────┐  ┌───────────────────────────────┐
│  Agent Studio 后端 (:8002)   │  │  Router Agent (:8000)         │
│  FastAPI BFF                 │  │  意图分类 → 调度 → 结果融合    │
│  ├─ 本地 SQLite (模板/工作流)│  │  ├─ UP 主管理                  │
│  └─ 代理 → agent-platform    │  │  ├─ 监控触发                   │
└──────────────┬───────────────┘  │  └─ 系统指标                   │
               │                  └──────────┬────────────────────┘
               ▼                             │
┌──────────────────────────────────┐         │
│  Agent Platform (:8001)          │         │
│  FastAPI + LangGraph             │         │
│  ├─ MCP Client                   │         │
│  ├─ 三层记忆 (短期/长期/反思)    │         │
│  ├─ 安全护栏 (输入/输出/SQL)     │         │
│  └─ 审计日志 (SQLite)            │         │
└──────────────────────────────────┘         │
                                             │
               ┌─────────────────────────────┘
               ▼
┌──────────────────────────────────────────────────────────────┐
│                    MCP 服务层 (SSE)                           │
│  Bilibili MCP (:9001)    RAG MCP (:9002)    SQL MCP (:9003)  │
└──────────────────────────────────────────────────────────────┘
```

注意 Agent Studio 的后端分成了两条路径：

- **本地路径**：Agent 模板、工作流、实例的 CRUD 操作直接读写本地 SQLite
- **代理路径**：工具状态、记忆数据、审计日志等请求转发给 agent-platform

这种双路径设计是 BFF 模式的核心——前端只跟一个后端打交道，后端帮你决定哪些请求自己处理、哪些请求需要转发。

| **TypeScript** | 5.6 | 类型安全，重构时编译器帮你兜底 |

这里有个值得说的点：我选了 `@xyflow/react` 而不是老版本的 `reactflow`。这是 ReactFlow 团队的新包名，API 更干净，性能也更好。工作流编辑器是整个项目里最复杂的页面，后面第二篇文章会详细讲。

### 4.2 后端技术栈

后端就三个核心技术：

| 技术 | 用途 | 为什么选它 |
|------|------|------------|
| **FastAPI** | HTTP 服务框架 | 异步原生支持、自动 OpenAPI 文档、Pydantic 校验 |
| **aiosqlite** | 异步 SQLite 驱动 | 不阻塞事件循环，和 FastAPI 完美配合 |
| **httpx** | 异步 HTTP 客户端 | 转发请求到 agent-platform，支持 HTTP/2 |

为什么后端这么"薄"？因为 Agent Studio 的后端本质上是一个 **BFF（Backend For Frontend）**，它不需要实现复杂的业务逻辑，只需要做两件事：

1. 管理本地数据（模板、工作流、实例）
2. 把其他请求代理给 agent-platform

越薄的 BFF 越好维护，这是设计原则。

## 五、BFF 代理架构与三跳代理链

这是整个架构里最有意思的部分。一个请求从浏览器出发到最终拿到数据，要经过**三跳**：

```
浏览器 ──第 1 跳──→ Vite Dev Server ──第 2 跳──→ FastAPI BFF ──第 3 跳──→ Agent Platform
 :5173                  :5173 proxy              :8002                    :8001
```

### 第 1 跳：Vite 代理

看看 `vite.config.ts` 的配置：

```typescript
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  server: {
    port: 5173,
    proxy: {
      '/api': {
        target: 'http://localhost:8002',
        changeOrigin: true,
      },
    },
  },
});
```

所有 `/api` 开头的请求，Vite 开发服务器都会透明地转发到后端的 8002 端口。前端代码里直接写 `fetch('/api/agents')` 就行，不用操心跨域问题。这个代理只在开发环境生效，生产环境会用 Nginx 来做。

### 第 2 跳：FastAPI BFF 的路由分发

BFF 后端收到请求后，根据路由决定是自己处理还是转发。看看 `main.py` 的路由注册：

```python
"""Agent Studio — FastAPI 管理后端"""
from contextlib import asynccontextmanager
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

from app.db import init_db, close_db
from app.routers import agents, workflows, tools, memory, dashboard, audit


@asynccontextmanager
async def lifespan(app: FastAPI):
    """启动/关闭时的资源管理"""
    await init_db()
    print("[OK] Agent Studio 后端启动完成")
    yield
    await close_db()
    print("[BYE] Agent Studio 后端已关闭")


app = FastAPI(
    title="Agent Studio",
    description="Agent 开发平台 — 可视化管理后端",
    version="1.0.0",
    lifespan=lifespan,
)

# CORS — 允许前端开发服务器访问
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# 注册路由
app.include_router(agents.router, prefix="/api/agents", tags=["Agent 管理"])
app.include_router(workflows.router, prefix="/api/workflows", tags=["工作流管理"])
app.include_router(tools.router, prefix="/api/tools", tags=["工具管理"])
app.include_router(memory.router, prefix="/api/memory", tags=["记忆系统"])
app.include_router(dashboard.router, prefix="/api/dashboard", tags=["仪表盘"])
app.include_router(audit.router, prefix="/api/audit", tags=["审计日志"])
```

六个路由模块，每个对应前端的一个页面。其中 `agents` 和 `workflows` 是本地 CRUD，其他四个会代理到 agent-platform。

### 第 3 跳：httpx 转发到 Agent Platform

以工具管理为例，BFF 收到 `/api/tools` 请求后，用 httpx 把请求转发到 agent-platform 的 `/api/mcp/tools`，拿到结果再返回给前端。整个过程是异步的，不会阻塞。

三跳代理链的好处是**解耦**：

- 前端不需要知道 agent-platform 的存在
- agent-platform 不需要知道有管理界面
- BFF 可以随时替换、扩展、甚至下线，不影响其他组件

前端路由定义在 `App.tsx` 里，非常清晰：

所有页面共享同一个 `Layout` 组件（侧边栏导航 + 顶栏），下面是七个页面的功能概览：

| 页面 | 路由 | 数据来源 | 核心功能 |
|------|------|----------|----------|
| **仪表盘** | `/` | 本地 + 代理 | 统计卡片 + Recharts 饼图 + 平台健康状态 |
| **Agent 市场** | `/agents` | 本地 SQLite | 4 个预置模板 + 创建实例 + 启停管理 |

| **记忆系统** | `/memory` | 代理 → agent-platform | 对话列表 + 详情查看 + 反思记录 |
| **审计日志** | `/audit` | 代理 → agent-platform | 事件时间线 + 类型筛选 + 链路追溯 |
| **安全治理** | `/security` | 代理 → agent-platform | 护栏策略展示 + 拦截事件记录 |

### 预置 Agent 模板

Agent 市场页面预置了 4 个模板，覆盖了最常见的 Agent 类型：

| 模板 | 类型 | 路由策略 | 适用场景 |
|------|------|----------|----------|
| 🧠 知识库助手 | hybrid | SQL + RAG + 融合 + 反思 | 通用知识问答 |
| 📊 数据分析师 | structured | 仅 SQL | 数据查询统计 |
| 💡 内容顾问 | semantic | 仅 RAG | 语义内容检索 |
| 🎯 多 Agent 协调器 | orchestrator | A2A + 全部 MCP | 复杂任务编排 |

### ReactFlow 工作流节点

工作流编辑器支持 7 种节点类型，对应 LangGraph 里的各个阶段：

| 节点类型 | 图标 | 功能 | 输入 → 输出 |
|----------|------|------|-------------|
| classify | 🧭 | 意图分类 | question → route_type, filters |
| route | 🔀 | 条件路由 | route_type → go_sql / go_rag / go_both |
| query_sql | 🗃️ | SQL 查询 | question, filters → sql_result |
| query_rag | 🔍 | RAG 检索 | question, filters → rag_result |
| query_both | ⚡ | 混合查询 | question, filters → sql_result + rag_result |
| merge | 🔗 | 结果融合 | sql_result + rag_result → final_answer |
| reflect | 💭 | 反思 | question + final_answer → (无输出) |

用户可以在画布上拖拽这些节点、连线、配置参数，然后保存为工作流模板。这比手写 LangGraph 代码直观多了。

## 七、本地 SQLite 存储设计

Agent Studio 没有引入 PostgreSQL 或 MySQL 这种重型数据库，而是用了 SQLite——因为作为一个管理控制台，它的数据量很小（几十条模板、几十条工作流），完全没必要上分布式数据库。

看看 `db.py` 里的表结构定义：

```python
async def init_db():
    """初始化数据库表 + 预置 Agent 模板"""
    db = await get_db()

    await db.executescript("""
        CREATE TABLE IF NOT EXISTS agent_templates (
            id TEXT PRIMARY KEY,
            name TEXT NOT NULL,
            description TEXT,
            type TEXT NOT NULL,
            icon TEXT DEFAULT '🤖',
            config TEXT DEFAULT '{}',
            workflow_id TEXT,
            created_at TEXT DEFAULT (datetime('now')),
            updated_at TEXT DEFAULT (datetime('now'))
        );

        CREATE TABLE IF NOT EXISTS workflows (
            id TEXT PRIMARY KEY,
            name TEXT NOT NULL,
            description TEXT,
            nodes TEXT DEFAULT '[]',
            edges TEXT DEFAULT '[]',
            config TEXT DEFAULT '{}',
            status TEXT DEFAULT 'draft',
            created_at TEXT DEFAULT (datetime('now')),
            updated_at TEXT DEFAULT (datetime('now'))
        );

        CREATE TABLE IF NOT EXISTS agent_instances (
            id TEXT PRIMARY KEY,
            template_id TEXT,
            name TEXT NOT NULL,
            config TEXT DEFAULT '{}',
            workflow_id TEXT,
            status TEXT DEFAULT 'active',
            created_at TEXT DEFAULT (datetime('now')),
            FOREIGN KEY (template_id) REFERENCES agent_templates(id)
        );
    """)
```

三张表，关系很清晰：

```
agent_templates (模板)
    │
    │ 1:N
    ▼
agent_instances (实例)  ──N:1──→  workflows (工作流)
```

### 设计要点

**1. JSON 字段存储复杂结构。** `workflows` 表的 `nodes` 和 `edges` 字段存的是 JSON 字符串。ReactFlow 编辑器输出的节点和边的数据结构很复杂，强行拆成关系表纯属自找麻烦。SQLite 存 JSON 文本完全没问题，取出来 `JSON.parse` 一下就行。

**2. 异步驱动不阻塞。** 用的是 `aiosqlite`，所有数据库操作都是 `await` 的，不会阻塞 FastAPI 的事件循环：

```python
async def get_db() -> aiosqlite.Connection:
    global _db
    if _db is None:
        os.makedirs(os.path.dirname(DB_PATH), exist_ok=True)
        _db = await aiosqlite.connect(DB_PATH)
        _db.row_factory = aiosqlite.Row
    return _db
```

注意 `_db.row_factory = aiosqlite.Row` 这行——它让查询结果可以用列名来访问（类似字典），而不是用下标，写起来舒服很多。

**3. 自动建表 + 种子数据。** `init_db()` 在应用启动时通过 `lifespan` 钩子自动调用。`CREATE TABLE IF NOT EXISTS` 保证幂等，模板数据只在首次运行时插入（`SELECT COUNT(*)` 检查）。这意味着新用户 clone 项目后直接启动就能看到预置的 4 个 Agent 模板。

### 数据库文件位置

```python
DB_PATH = os.path.join(os.path.dirname(__file__), "..", "data", "agent_studio.db")
```

数据库文件在 `backend/data/agent_studio.db`，`os.makedirs` 保证 `data` 目录自动创建。这个文件加到 `.gitignore` 里就行，不提交到仓库。

## 八、中文优先的设计理念

这个项目从第一天起就是面向中文用户的。不是简单的 i18n 切换，而是从数据层到 UI 层全链路中文：

| 层面 | 体现 |
|------|------|
| **数据库** | 模板名称是"知识库助手"不是"Knowledge Assistant" |
| **API 描述** | FastAPI 的 tags 是"Agent 管理""工作流管理" |
| **UI 文案** | 所有按钮、提示、标题都是中文 |
| **配置字段** | Agent 的 config 里的 features 是"意图分类""结果融合" |
| **文档** | 代码注释和架构文档都是中文 |

看看预置模板的中文配置：

```python
{
    "id": "knowledge-assistant",
    "name": "知识库助手",
    "description": "综合型知识库问答助手，支持结构化数据查询和语义检索，"
                   "自动路由到最佳查询方式。适合 B 站视频知识库、"
                   "企业文档库等场景。",
    "type": "hybrid",
    "icon": "🧠",
    "config": json.dumps({
        "route_types": ["structured", "semantic", "hybrid"],
        "mcp_servers": ["sql", "rag"],
        "features": ["意图分类", "SQL 查询", "RAG 检索", "结果融合", "反思"],
    }),
},
```

这种"中文优先"的策略让我在开发时不需要在脑子里做中英文切换，写起来更自然，debug 时看日志也更直观。当然，如果未来要做国际化，把文案抽成 i18n key 也不难——但目前没有这个需求，就不提前优化了。

## 总结

这篇文章介绍了 Agent Studio 的全貌：

- **为什么做**：Agent 系统越来越复杂，需要一个可视化的管理界面来降低运维和调试的门槛
- **在系统中的位置**：三件套的第三环，管理控制台层，通过 BFF 代理与 Agent 中枢通信
- **架构设计**：前端 React SPA → Vite 代理 → FastAPI BFF → Agent Platform，三跳代理链实现松耦合
- **技术栈**：React 18 + TypeScript + Vite + Tailwind + ReactFlow + Recharts（前端），FastAPI + aiosqlite + httpx（后端）
- **存储方案**：本地 SQLite 存管理数据（模板、工作流、实例），其他数据通过代理获取
- **设计理念**：中文优先、BFF 薄层、按需代理

下一篇文章，我会深入前端实战，手把手讲解如何用 ReactFlow 构建工作流编辑器——包括自定义节点的设计、拖拽交互的实现、以及如何把 ReactFlow 的状态和后端 SQLite 同步起来。如果你正在做类似的可视化编辑器，那篇一定不要错过。
