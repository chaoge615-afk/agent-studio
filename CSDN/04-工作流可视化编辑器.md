## 前言

在前面三篇文章中，我们分别聊了 Agent Studio 的整体架构、后端的 FastAPI 实现，以及 MCP 服务的管理。但说实话，不管后端写得多优雅，如果用户只能通过写 JSON 或者敲命令行来编排工作流，那这个平台的门槛就太高了。

所以这一篇我们要解决一个核心问题：**怎么让用户像搭积木一样，在浏览器里拖拖拽拽就能编排出一套完整的 AI Agent 工作流？** 答案是可视化编辑器。我们选用了 ReactFlow，结合自定义的 8 种节点类型，做出了一套还算好用的搭建界面。

这篇文章我会从"为什么要做可视化"开始，一路讲到 ReactFlow 集成、CustomNode 的 8 种变体、Handle 设计、页面布局、预置模板、CRUD 操作，最后聊聊前端节点跟 LangGraph 执行引擎的对应关系。代码全部来自项目真实源码，不掺水。

## 一、为什么需要可视化编辑器？

Agent 编排本质上是在定义一个有向图（DAG）。用代码写没问题，但三个痛点很明显：**不直观**（7、8 个节点加条件分支，光看代码很难看出全貌）、**调试困难**（数据在哪条边上断了？）、**非开发者用不了**（产品经理也想参与 Agent 设计）。

所以我们做了可视化编辑器，拖拽节点、连线、调参数，所见即所得：

```
+----------+     +----------+     +-----------+
| 意图分类  | --> | 条件路由  | --> | SQL 查询  |
+----------+     +----------+     +-----------+
                      |                |
                      v                v
                 +-----------+    +----------+
                 | RAG 检索  |    | 结果融合  |
                 +-----------+    +----------+
                                      |
                                      v
                                 +----------+
                                 |   反思    |
                                 +----------+
```

## 二、技术选型：为什么是 ReactFlow

市面上做流程可视化的库不少，我考察过的几个：

| 库 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| ReactFlow (@xyflow/react) | React 原生、API 清晰、社区活跃 | 复杂布局需自己写 | ✅ 选用 |
| X6 (AntV) | 功能丰富 | 体积大、React 集成一般 | ❌ |
| jsPlumb | 老牌、灵活 | API 老旧、不 React | ❌ |
| LogicFlow | 国产、流程图专精 | 社区较小 | 备选 |

最终选了 ReactFlow（现在包名 `@xyflow/react`），因为它跟 React 的契合度最高，自定义节点 API 干净，而且 `MiniMap`、`Controls`、`Background` 都是开箱即用的。

```tsx
import { ReactFlow, Background, Controls, MiniMap, addEdge,
  useNodesState, useEdgesState, Panel } from '@xyflow/react';
import '@xyflow/react/dist/style.css';
```

## 三、页面整体布局

工作流页面是经典的双栏结构：左侧深色侧边栏，右侧 ReactFlow 画布。

```
+------------------+--------------------------------------------+
|  工作流列表       |  [名称] [保存] [导出] [删除]                |
|  ├─ 混合查询      |                                            |
|  ├─ RAG 检索     |         ReactFlow 画布                      |
|  └─ 数据分析      |    (节点 + 连线 + MiniMap + Controls)       |
|  -- 从模板创建 -- |                                            |
|  节点类型面板     |                                            |
+------------------+--------------------------------------------+
```

核心布局代码：

```tsx
<div className="h-[calc(100vh-3rem)] flex gap-4 page-enter">
  {/* 左侧面板 — 深色背景 */}
  <div className="w-64 flex-shrink-0 flex flex-col gap-3 overflow-y-auto">
    <div className="bg-gray-900 rounded-xl p-4">{/* 工作流列表 */}</div>
    <div className="bg-gray-900 rounded-xl p-4">{/* 节点类型面板 */}</div>
  </div>

  {/* ReactFlow 画布 */}
  <div className="flex-1 bg-white rounded-xl border border-gray-200/60 overflow-hidden shadow-card">
    {selectedId ? (
      <ReactFlow nodes={nodes} edges={edges} nodeTypes={nodeTypes} fitView>
        <Background />
        <Controls />
        <MiniMap />
        <Panel position="top-left">{/* 工具栏 */}</Panel>
      </ReactFlow>
    ) : (
      <div className="empty-state">选择或创建工作流</div>
    )}
  </div>
</div>
```

左侧 `bg-gray-900` 深色背景与右侧白色画布形成对比，`w-64` 固定侧边栏 256px，剩余空间全部留给画布。

## 四、CustomNode：八种节点类型的设计

这是整个编辑器最核心的部分。我们的 Agent 工作流有 8 种节点类型，每种都有独立的图标、颜色和语义。

### 4.1 八种节点一览

| 节点类型 | 图标 | 用途 | 色系 |
|---------|------|------|------|
| `classify` | 🧭 | 意图分类，用 LLM 判断问题类型 | 紫色 |
| `route` | 🔀 | 条件路由，根据分类结果分发 | 紫罗兰 |
| `query_sql` | 🗃️ | SQL 查询，调用 Text-to-SQL | 靛蓝 |
| `query_rag` | 🔍 | RAG 检索，语义搜索 | 青色 |
| `query_both` | ⚡ | 混合查询，SQL + RAG 并行 | 蓝色 |
| `merge` | 🔗 | 结果融合，合并多路结果 | 翡翠绿 |
| `reflect` | 💭 | 反思，分析质量存入记忆 | 琥珀 |
| `custom` | ⚙️ | 自定义节点 | 灰色 |

### 4.2 样式映射与完整组件

每种节点的视觉样式通过一个 Record 管理，样式和逻辑解耦，新增节点类型只需加一行：

```tsx
import { memo } from 'react';
import { Handle, Position, NodeProps } from '@xyflow/react';

const nodeStyles: Record<string, { icon: string; borderColor: string; bgGradient: string }> = {
  classify:   { icon: '🧭', borderColor: 'border-purple-300',  bgGradient: 'bg-gradient-to-br from-purple-50 to-violet-50' },
  route:      { icon: '🔀', borderColor: 'border-violet-300',  bgGradient: 'bg-gradient-to-br from-violet-50 to-fuchsia-50' },
  query_sql:  { icon: '🗃️', borderColor: 'border-indigo-300',  bgGradient: 'bg-gradient-to-br from-indigo-50 to-blue-50' },
  query_rag:  { icon: '🔍', borderColor: 'border-cyan-300',    bgGradient: 'bg-gradient-to-br from-cyan-50 to-sky-50' },
  query_both: { icon: '⚡', borderColor: 'border-blue-300',    bgGradient: 'bg-gradient-to-br from-blue-50 to-indigo-50' },
  merge:      { icon: '🔗', borderColor: 'border-emerald-300', bgGradient: 'bg-gradient-to-br from-emerald-50 to-teal-50' },
  reflect:    { icon: '💭', borderColor: 'border-amber-300',   bgGradient: 'bg-gradient-to-br from-amber-50 to-orange-50' },
  custom:     { icon: '⚙️', borderColor: 'border-gray-300',    bgGradient: 'bg-gradient-to-br from-gray-50 to-slate-50' },
};

function CustomNode({ data, selected }: NodeProps) {
  const style = nodeStyles[data.nodeType as string] || nodeStyles.custom;

  return (
    <div className={`px-4 py-3 rounded-xl border-2 ${style.borderColor} ${style.bgGradient} ${
      selected ? 'ring-2 ring-indigo-500/50 shadow-lg' : 'shadow-md'
    } min-w-[150px] transition-all duration-200`}>
      <Handle type="target" position={Position.Top}
        className="!w-3 !h-3 !bg-indigo-400 !border-2 !border-white !shadow-sm" />
      <div className="flex items-center gap-2.5">
        <span className="text-xl">{style.icon}</span>
        <div>
          <div className="font-semibold text-sm text-gray-900">{data.label as string}</div>
          {data.description ? (
            <div className="text-xs text-gray-500 mt-0.5">{String(data.description)}</div>
          ) : null}
        </div>
      </div>
      <Handle type="source" position={Position.Bottom}
        className="!w-3 !h-3 !bg-indigo-400 !border-2 !border-white !shadow-sm" />
    </div>
  );
}

export default memo(CustomNode);
```

### 4.3 Handle 连接点与 memo()

每个节点有两个 Handle：**顶部 target**（接收输入）和**底部 source**（输出到下一个节点），数据流方向是从上到下。

```
     ●  ← target Handle (Position.Top)
   +------+
   | 节点  |
   +------+
     ●  ← source Handle (Position.Bottom)
```

Handle 样式用了 Tailwind 的 `!important` 前缀（`!w-3`）来覆盖 ReactFlow 默认样式。靛蓝色小圆点加白色描边，在画布上辨识度很高。

注意最后的 `export default memo(CustomNode)`。ReactFlow 拖拽、缩放时会频繁触发重渲染，20 个节点如果不做 memo 优化，每次操作全部重渲染会很卡。`memo()` 确保只有 `data` 或 `selected` 真正变化时才重渲染，实测流畅度提升明显。

## 五、节点类型面板与添加节点

左侧侧边栏下半部分是"节点类型面板"，数据来自后端 API `/workflows/node-types`，不是前端硬编码的。

### 5.1 后端返回的节点元信息

后端为每种节点返回完整元信息，包括 inputs、outputs 和 config_fields：

```json
{
  "type": "query_rag", "label": "RAG 检索", "icon": "🔍", "category": "query",
  "description": "通过 MCP 调用 RAG 服务执行语义检索",
  "inputs": ["question", "filters"],
  "outputs": ["rag_result"],
  "config_fields": [
    { "key": "server", "label": "MCP Server", "type": "select", "default": "rag" },
    { "key": "top_k", "label": "返回条数", "type": "number", "default": 5 }
  ]
}
```

前端加载：

```tsx
const loadNodeTypes = async () => {
  const res = await api.get<{ node_types: NodeType[] }>('/workflows/node-types');
  setNodeTypes_(res.node_types || []);
};
```

### 5.2 点击添加节点

用户点击面板按钮，画布上就出现一个新节点：

```tsx
const addNode = (nodeType: NodeType) => {
  const id = `node_${Date.now()}`;
  const newNode: Node = {
    id, type: 'custom',
    position: { x: 250 + Math.random() * 100, y: 100 + nodes.length * 120 },
    data: { label: nodeType.label, nodeType: nodeType.type,
            description: nodeType.description, config: {} },
  };
  setNodes((nds) => [...nds, newNode]);
};
```

有个小细节：`position` 加了 `Math.random() * 100` 随机偏移，避免连续添加的节点完全重叠，用户能立刻看到"哦，新加了一个"。

## 六、三个预置工作流模板

为降低使用门槛，我们预置了三个常用模板，用户点一下就能创建完整的节点和连线。

| 模板 ID | 名称 | 节点数 | 适用场景 |
|---------|------|-------|---------|
| `standard-hybrid` | 标准混合查询流程 | 7 | 完整的意图分类→路由→查询→融合→反思 |
| `simple-rag` | 简单 RAG 检索流程 | 2 | 纯问答场景 |
| `data-pipeline` | 数据分析流程 | 3 | 结构化数据查询 |

**标准混合查询流程**是最完整的模板：

```
🧭 意图分类 → 🔀 条件路由 → (🗃️ SQL / 🔍 RAG / ⚡ 混合) → 🔗 结果融合 → 💭 反思
```

后端模板定义在 `DEFAULT_WORKFLOWS` 字典中。以 standard-hybrid 为例，每个节点都带有精确坐标，方便 ReactFlow 直接渲染：

```python
DEFAULT_WORKFLOWS = {
    "standard-hybrid": {
        "name": "标准混合查询流程",
        "nodes": [
            {"id": "classify", "type": "classify", "label": "意图分类",
             "position": {"x": 250, "y": 0}},
            {"id": "route", "type": "route", "label": "条件路由",
             "position": {"x": 250, "y": 120}},
            {"id": "query_sql", "type": "query_sql", "label": "SQL 查询",
             "position": {"x": 50, "y": 240}},
            # ... 共 7 个节点
        ],
        "edges": [
            {"id": "e1", "source": "classify", "target": "route"},
            {"id": "e2", "source": "route", "target": "query_sql", "label": "structured"},
            {"id": "e3", "source": "route", "target": "query_rag", "label": "semantic"},
            # ... 共 8 条边
        ],
    },
}
```

前端一键创建并自动打开：

```tsx
const createFromTemplate = async (templateId: string) => {
  const wf = await api.post<any>(`/workflows/from-template/${templateId}`, {});
  loadWorkflows();
  loadWorkflow(wf.id);  // 自动打开
};
```

## 七、CRUD 操作：前后端联动

工作流的增删改查是编辑器的骨架。重点看前后端之间的数据格式转换。

### 7.1 读取 — 加载工作流

加载时要把后端的"业务节点"转成 ReactFlow 的"画布节点"格式。关键是所有节点的 `type` 统一为 `'custom'`，真正的类型信息放在 `data.nodeType` 里：

```tsx
const loadWorkflow = async (id: string) => {
  const wf = await api.get<any>(`/workflows/${id}`);
  setSelectedId(id);
  setWorkflowName(wf.name);
  const rfNodes: Node[] = (wf.nodes || []).map((n: any) => ({
    id: n.id, type: 'custom', position: n.position || { x: 0, y: 0 },
    data: { label: n.label, nodeType: n.type,
            description: n.description || '', config: n.config || {} },
  }));
  const rfEdges: Edge[] = (wf.edges || []).map((e: any) => ({
    id: e.id, source: e.source, target: e.target, label: e.label || '',
    animated: true, style: { stroke: '#6366f1', strokeWidth: 2 },
  }));
  setNodes(rfNodes); setEdges(rfEdges);
};
```

### 7.2 保存 — 反向转换

```tsx
const saveWorkflow = async () => {
  if (!selectedId) return;
  const nodesData = nodes.map((n) => ({
    id: n.id, type: n.data?.nodeType || 'custom',
    label: n.data?.label || n.id, config: n.data?.config || {}, position: n.position,
  }));
  const edgesData = edges.map((e) => ({
    id: e.id, source: e.source, target: e.target, label: e.label?.toString() || '',
  }));
  await api.put(`/workflows/${selectedId}`, { name: workflowName, nodes: nodesData, edges: edgesData });
};
```

保存时做反向转换：从 ReactFlow 格式转回业务格式。`type` 字段从 `data.nodeType` 取出，不是 ReactFlow 的 `'custom'`。

### 7.3 删除

```tsx
const deleteWorkflow = async () => {
  if (!selectedId || !confirm('确定要删除这个工作流吗？')) return;
  await api.delete(`/workflows/${selectedId}`);
  setSelectedId(null);
  setNodes([]); setEdges([]);
  loadWorkflows();
};
```

### 7.4 后端的灵活更新

后端更新接口只更新传过来的字段，动态拼 SQL，减少不必要的网络传输：

```python
@router.put("/{workflow_id}")
async def update_workflow(workflow_id: str, data: WorkflowUpdate):
    updates = {}
    if data.name is not None:  updates["name"] = data.name
    if data.nodes is not None:
        updates["nodes"] = json.dumps([n.model_dump() for n in data.nodes], ensure_ascii=False)
    if data.edges is not None:
        updates["edges"] = json.dumps([e.model_dump() for e in data.edges], ensure_ascii=False)
    updates["updated_at"] = datetime.now().isoformat()
    set_clause = ", ".join(f"{k} = ?" for k in updates.keys())
    await db.execute(f"UPDATE workflows SET {set_clause} WHERE id = ?", values)
```

## 八、JSON 导出与 MiniMap 辅助

### 8.1 一键导出 JSON

```tsx
const exportJSON = () => {
  const data = { nodes: nodes.map(n => ({...n})), edges: edges.map(e => ({...e})) };
  const blob = new Blob([JSON.stringify(data, null, 2)], { type: 'application/json' });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `${workflowName || 'workflow'}.json`;
  a.click();
};
```

导出的 JSON 包含所有节点和连线的完整信息（位置、类型、配置），可用于版本管理或跨环境迁移。

### 8.2 MiniMap、Controls 与连线动画

ReactFlow 自带的三个辅助组件直接嵌入画布：

- **Background**：点阵网格背景，让用户感知"在移动画布"。
- **Controls**：左下角缩放/居中按钮组。
- **MiniMap**：右下角缩略图，节点多时快速定位。

新建连线时自动添加靛蓝色动画：

```tsx
const onConnect = useCallback(
  (params: Connection) =>
    setEdges((eds) => addEdge(
      { ...params, animated: true, style: { stroke: '#6366f1', strokeWidth: 2 } }, eds
    )),
  [setEdges]
);
```

`animated: true` 让连线有流动虚线动画，视觉上暗示"数据在流动"，比静态实线更有表现力。

## 九、节点如何映射到 LangGraph

前端 8 种节点类型不只是视觉把戏——它们跟后端 LangGraph 执行引擎有严格对应关系。

### 9.1 映射关系

```
前端节点类型          LangGraph 对应
─────────────────    ──────────────────────────
classify          →  graph.add_node("classify", classify_intent)
route             →  graph.add_conditional_edges("route", ...)
query_sql         →  graph.add_node("query_sql", run_sql_via_mcp)
query_rag         →  graph.add_node("query_rag", run_rag_via_mcp)
query_both        →  graph.add_node("query_both", run_parallel)
merge             →  graph.add_node("merge", merge_results)
reflect           →  graph.add_node("reflect", reflect_and_store)
custom            →  graph.add_node("custom", user_defined_fn)
```

### 9.2 条件路由的实现

`route` 节点在前端只是一个普通节点，在后端对应 LangGraph 的 **conditional edges**：

```
         route
        /  |  \
  structured | hybrid
      /   semantic  \
     v      v        v
  query_sql query_rag query_both
```

后端根据 classify 输出的 `route_type` 决定分支：

```python
def decide_route(state: AgentState) -> str:
    route_type = state.get("route_type", "semantic")
    if route_type == "structured":   return "query_sql"
    elif route_type == "hybrid":     return "query_both"
    else:                            return "query_rag"

graph.add_conditional_edges("route", decide_route)
```

### 9.3 数据流：节点的输入输出

后端 `/node-types` 接口为每种节点声明了 `inputs` 和 `outputs`：

| 节点 | 输入 | 输出 |
|------|------|------|
| classify | question, messages, memory_context | route_type, filters |
| route | route_type | go_sql, go_rag, go_both |
| query_sql | question, filters | sql_result |
| query_rag | question, filters | rag_result |
| query_both | question, filters | sql_result, rag_result |
| merge | sql_result, rag_result | final_answer, sources |
| reflect | question, final_answer | (无，写入记忆) |

这个声明目前用于前端展示，后续会基于它做自动化数据流校验。
