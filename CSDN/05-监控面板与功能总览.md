## 前言

这是 Agent Studio 系列的收官之作。前四篇文章我们分别聊了项目架构、后端 API、前端组件体系和 LangGraph 引擎编排，算是把"怎么造"这件事拆解得差不多了。但一个平台造出来之后，真正让运维和使用者安心的，是——**能不能一眼看清它在干什么**。

这篇文章我带你逛遍整个前端：从仪表盘的全局概览，到 Agent 市场的模板与实例管理；从工作流编辑器的可视化编排，到 MCP 工具的实时连接状态；从记忆系统的对话回溯，到审计日志的全链路追踪和安全治理的拦截记录。七个页面，覆盖了 Agent 从创建、运行、调用工具、记忆对话，到安全审计的完整生命周期。

说实话，这些页面加起来也就一千多行 React 代码，但它们承载的信息密度非常高。我会挑每个页面最核心的代码片段，配合架构图和表格，让你快速理解设计思路。如果你是这个系列从头跟到尾的读者，这篇就是最后一块拼图。

## 一、总览：七大页面构成与数据流

先放一张全局架构图，让你对七个页面的数据来源有个整体认知：

```

                          │     React Router          │
                          │  /  /agents /workflows    │
                          │  /tools /memory /audit    │
                          │  /security                │
                          └────────┬─────────────────┘
                                   │ fetch
                                   ▼
                          ┌──────────────────────────┐
                          │   API Client (api.ts)     │
                          │   /dashboard/stats        │
                          │   /agents, /workflows     │
                          │   /tools, /memory/*       │
                          │   /audit/logs, /audit/event-types │
                          └────────┬─────────────────┘
                                   │ HTTP
                                   ▼
                 ┌─────────────────────────────────────┐
                 │      FastAPI Backend (:8002)         │
                 │  routers: agents / workflows / tools │
                 │  dashboard / memory / audit          │
                 └────────┬────────────────────────────┘
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
         ┌────────┐  ┌────────┐  ┌──────────────┐
         │ SQLite │  │  MCP   │  │ LangGraph    │
         │ Audit  │  │Servers │  │ Agent Engine │
```

七个页面对应的职责分工：

| 页面 | 路由 | 核心职责 | 数据来源 |
|------|------|----------|----------|
| Dashboard | `/` | 全局统计、事件分布、平台健康 | `/dashboard/stats` |
| Agents | `/agents` | 模板浏览、实例创建与启停 | `/agents`, `/agents/instances/*` |
| Workflows | `/workflows` | 工作流可视化编辑与管理 | `/workflows`, `/workflows/templates` |
| Tools | `/tools` | MCP 连接状态、工具参数展示 | `/tools` |
| Memory | `/memory` | 对话线程、Q&A 回溯 | `/memory/conversations`, `/memory/stats` |
| Audit | `/audit` | 审计日志过滤与时间线 | `/audit/logs`, `/audit/event-types` |
| Security | `/security` | Guardrail 策略与拦截记录 | `/audit/logs?event_type=guardrail` |

## 二、仪表盘（Dashboard）——平台健康一目了然

### 2.1 四张统计卡片

Dashboard 页面打开，第一眼就是四张 StatCard，分别展示 Agent 模板数、运行中实例数、工作流数和审计事件总数。数据来自一个聚合接口：

```typescript
interface DashboardStats {
  templates: number; instances: number; workflows: number;
  audit: { total_events: number; event_types: Record<string, number> };
  health: { status: string }; recent_agents: any[];
}
```

数据通过单次请求聚合获取：

```typescript
const data = await api.get<DashboardStats>('/dashboard/stats');
setStats(data);
```

卡片的渲染使用了 stagger 动画，依次淡入：

```tsx
<div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-4 gap-5">
  <div className="opacity-0 animate-fade-in-up stagger-1">
    <StatCard icon={<Bot />} label="Agent 模板" value={stats?.templates ?? 0} color="blue" />
  </div>
  <div className="opacity-0 animate-fade-in-up stagger-2">
    <StatCard icon={<Activity />} label="运行中实例" value={stats?.instances ?? 0} color="green" />
  </div>
  {/* stagger-3: Workflow/工作流, stagger-4: Wrench/审计事件 — 结构相同 */}
</div>
```

四张卡片采用响应式网格：移动端单列、平板双列、桌面四列。每张卡片的 `stagger-N` class 控制动画延迟，形成从左到右的波浪感。

### 2.2 Recharts 甜甜圈图：审计事件分布

卡片下方左侧是一个甜甜圈图，数据源是 `stats.audit.event_types`——一个 `{ [eventType]: count }` 的字典。我把它转成 Recharts 需要的数组格式：

```tsx
const COLORS = ['#6366f1', '#8b5cf6', '#10b981', '#f59e0b', '#ef4444', '#06b6d4'];
const pieData = Object.entries(eventTypes).map(([name, value]) => ({ name, value }));

<ResponsiveContainer width="100%" height={260}>
  <PieChart>
    <Pie data={pieData} cx="50%" cy="50%"
         outerRadius={85} innerRadius={45}
         dataKey="value" label={({ name, value }) => `${name}: ${value}`}
         strokeWidth={2} stroke="#fff">
      {pieData.map((_, i) => <Cell key={i} fill={COLORS[i % COLORS.length]} />)}
    </Pie>
    <Tooltip contentStyle={{ borderRadius: '10px', fontSize: '13px' }} />
  </PieChart>
</ResponsiveContainer>
```

`innerRadius={45}` 配合 `outerRadius={85}` 形成甜甜圈效果，中间留白让视觉更透气。六种颜色循环使用，覆盖了 `chat_request`、`routing`、`mcp_call`、`answer`、`guardrail`、`error` 等事件类型。

### 2.3 平台状态面板

右侧是平台状态卡片，显示后端服务连接情况以及最近创建的 Agent：

```tsx
<span className="text-sm font-medium text-gray-700">agent-platform</span>
<span className={stats?.health?.status === 'healthy' ? 'status-active' : 'status-error'}>
  {stats?.health?.status === 'healthy' ? '运行中' : '未连接'}
</span>
```

`status-active` 和 `status-error` 是前面文章定义的全局 CSS class，分别是绿色和红色小圆点加文字。Dashboard 只是快照视图，每次加载时拉取最新状态即可。

## 三、Agent 市场——模板网格与实例管理

Agents 页面分为三个区域：顶部标题栏 + 模板网格 + 实例列表。数据加载用了 `Promise.all` 并行请求：

```typescript
const loadData = async () => {
  const [agentsRes, instancesRes] = await Promise.all([
    api.get<{ count: number; agents: AgentTemplate[] }>('/agents'),
    api.get<{ count: number; instances: AgentInstance[] }>('/agents/instances/list'),
  ]);
  setTemplates(agentsRes.agents);
  setInstances(instancesRes.instances);
};
```

### 3.1 模板卡片网格

模板用两列网格展示，每张卡片包含图标、名称、类型标签、描述和功能特性标签：

```tsx
<div className="grid grid-cols-1 md:grid-cols-2 gap-4">
  {templates.map((template) => (
    <div key={template.id} className="card-hover">
      <div className="flex items-start gap-4">
        <div className="w-12 h-12 rounded-xl bg-gradient-to-br from-indigo-500/10 to-purple-500/10">
          <span className="text-2xl">{template.icon}</span>
        </div>
        <div className="flex-1 min-w-0">
          <h3 className="font-semibold">{template.name}</h3>
          <span className="badge badge-blue">{template.type}</span>
          <p className="text-sm text-gray-500">{template.description}</p>
          {template.config?.features && (
            <div className="flex flex-wrap gap-1.5 mt-3">
              {template.config.features.map((f: string) => (
                <span key={f} className="badge badge-purple">{f}</span>
              ))}
            </div>
          )}
        </div>
      </div>
    </div>
  ))}
</div>
```

### 3.2 创建实例模态框

点击"创建实例"按钮会弹出一个模态框，采用固定定位 + 毛玻璃遮罩的经典方案：

```tsx
{showCreate && (
  <div className="fixed inset-0 z-50 flex items-center justify-center">
    <div className="absolute inset-0 bg-black/30 backdrop-blur-sm"
         onClick={() => setShowCreate(false)} />
    <div className="relative bg-white rounded-2xl shadow-elevated p-6 w-full max-w-md animate-fade-in-up">
      <select className="input" value={selectedTemplate}
              onChange={(e) => setSelectedTemplate(e.target.value)}>
        {templates.map((t) => <option key={t.id} value={t.id}>{t.icon} {t.name}</option>)}
      </select>
      <input className="input" placeholder="我的 Agent" value={newInstanceName}
             onChange={(e) => setNewInstanceName(e.target.value)}
             onKeyDown={(e) => e.key === 'Enter' && createInstance()} />
    </div>
  </div>
)}
```

`backdrop-blur-sm` 实现毛玻璃效果，`animate-fade-in-up` 让模态框从底部弹入。输入框绑定了 `onKeyDown`，按回车直接提交。

### 3.3 实例列表与启停操作

实例列表每行展示图标、名称、模板来源、创建日期，以及启停/删除按钮：

```tsx
const toggleStatus = async (id: string, currentStatus: string) => {
  await api.put(`/agents/instances/${id}`, { status: currentStatus === 'active' ? 'stopped' : 'active' });
  loadData();
};
```

启停就是一个 PUT 请求，在后端切换状态。图标会根据当前状态变化：运行中显示方块（停止），已停止显示三角（启动）。删除前有 `confirm` 二次确认，防止误操作。

## 四、工具管理——MCP Server 状态与工具清单

Tools 页面是我个人觉得"信息密度设计最合理"的一个页面。结构非常清晰：

```
┌────────────────────────────────────────────┐
│  MCP 连接状态                               │
│  ● agent-platform  已连接                   │
├────────────────────────────────────────────┤
│  🔧 Server: filesystem                     │
│  3 个工具                                   │
│  ├─ read_file  [MCP]                       │
│  │   读取文件内容                             │
│  │   [path] [encoding]                      │
│  ├─ write_file [MCP]                       │
│  │   写入文件内容                             │
│  │   [path] [content]                       │
│  └─ list_dir   [MCP]                       │
│      列出目录内容                             │
│      [path]                                 │
└────────────────────────────────────────────┘
```

连接状态用绿色 `CheckCircle` 或红色 `XCircle` 图标直观展示：

```tsx
<div className="card">
  <div className="flex items-center gap-3">
    {connected ? <CheckCircle className="w-5 h-5 text-emerald-500" />
               : <XCircle className="w-5 h-5 text-red-400" />}
    <span className="font-medium text-gray-900">agent-platform</span>
    <span className={connected ? 'status-active' : 'status-error'}>
      {connected ? '已连接' : '未连接'}
    </span>
  </div>
</div>
```

每个 Server 下挂载工具列表，工具的参数直接解析自 `inputSchema.properties`：

```tsx
{server.tools.map((tool) => (
  <div key={tool.name} className="list-item">
    <div className="flex items-center justify-between">
      <span className="font-medium text-sm text-gray-900">{tool.name}</span>
      <span className="badge badge-blue">MCP</span>
    </div>
    {tool.description && <p className="text-xs text-gray-500 mt-1">{tool.description}</p>}
    {tool.inputSchema?.properties && (
      <div className="mt-2 flex flex-wrap gap-1.5">
        {Object.keys(tool.inputSchema.properties).map((param) => (
          <span key={param} className="badge badge-purple">{param}</span>
        ))}
      </div>
    )}
  </div>
))}
```

这段代码做了一个很讨巧的事：直接从 MCP 协议返回的 JSON Schema 中提取参数名，用紫色标签展示。不需要后端额外维护一份参数描述——MCP 协议本身就是自描述的。

## 六、审计日志——过滤器与时间线

审计页面解决的核心问题是：**当 Agent 做了一件事，我怎么回溯它的完整链路？**

页面提供两个维度的过滤器——按事件类型和按线程 ID：

```tsx
<div className="flex gap-4 items-end">
  <div className="flex-1">
    <label className="block text-xs font-medium text-gray-500 mb-1.5">事件类型</label>
    <select className="input" value={filterType} onChange={(e) => setFilterType(e.target.value)}>
      <option value="">全部类型</option>
      {eventTypes.map((t) => <option key={t.type} value={t.type}>{t.icon} {t.label}</option>)}
    </select>
  </div>
  <div className="flex-1">
    <label className="block text-xs font-medium text-gray-500 mb-1.5">线程 ID</label>
    <input className="input" placeholder="default" value={filterThread}
           onChange={(e) => setFilterThread(e.target.value)} />
  </div>
</div>
```

事件类型从后端 `/audit/event-types` 动态获取，每种类型配有图标和标签。过滤请求：

```typescript
const loadFiltered = async () => {
  const params: any = { limit: 100 };
  if (filterType) params.event_type = filterType;
  if (filterThread) params.thread_id = filterThread;
  const res = await auditApi.getLogs(params);
  setLogs(res.logs || []);
};
```

时间线用 CSS 的 `timeline-item` class 渲染，左侧有一条竖线连接各事件。每条日志展示事件图标、类型标签、线程 ID、耗时和时间戳：

```tsx
{logs.map((log) => (
  <div key={log.id} className="timeline-item">
    <div className="list-item">
      <div className="flex items-center justify-between mb-1.5">
        <div className="flex items-center gap-2">
          <span>{getEventIcon(log.event_type)}</span>
          <span className={`badge ${getEventBadge(log.event_type)}`}>{log.event_type}</span>
          <span className="text-[11px] text-gray-400 font-mono">{log.thread_id}</span>
        </div>
        {log.duration_ms != null && (
          <span className="text-xs text-gray-400 font-mono">{log.duration_ms.toFixed(0)}ms</span>
        )}
      </div>
    </div>
  </div>
))}
```

`getEventBadge` 函数根据事件类型返回对应的颜色 class，让时间线在视觉上就能区分正常请求（蓝色）、路由（紫色）、工具调用（黄色）、回答（绿色）和安全拦截（红色）。

## 七、安全治理——Guardrail 卡片与拦截日志

Security 页面分为两部分：上方三张策略卡片，下方拦截记录列表。

三张策略卡片分别覆盖输入过滤、输出过滤和 SQL 安全检查：

```typescript
const policyCards = [
  { title: '输入过滤', gradient: 'from-emerald-500/10 to-teal-500/10', iconColor: 'text-emerald-600',
    items: ['敏感词检测', 'Prompt 注入防护', '长度限制检查'] },
  { title: '输出过滤', gradient: 'from-indigo-500/10 to-blue-500/10', iconColor: 'text-indigo-600',
    items: ['身份证号脱敏', '银行卡号脱敏', '手机号脱敏'] },
  { title: 'SQL 安全检查', gradient: 'from-amber-500/10 to-orange-500/10', iconColor: 'text-amber-600',
    icon: AlertTriangle, items: ['拦截 DELETE/UPDATE', '拦截 DROP/TRUNCATE', '拦截 INSERT/ALTER'] },
];
```

每张卡片用渐变背景区分类型，每个策略项前有绿色勾号表示"已启用"。这是**静态声明式 UI**——Guardrail 规则在 agent-platform 后端硬编码（参见 agent-platform 系列第六篇《安全治理与审计》），前端只展示启用状态。

拦截记录从审计日志中过滤 `event_type=guardrail` 的事件：

```typescript
const loadGuardrailEvents = async () => {
  const result = await auditApi.getLogs({ event_type: 'guardrail', limit: 50 });
  const events = result.logs.map((log: AuditEntry) => ({
    timestamp: log.timestamp,
    action: log.event_data?.action || 'unknown',
    reason: log.event_data?.reason || '',
    severity: log.event_data?.severity || 'info',
  }));
  setGuardrailEvents(events);
};
```

严重等级用不同颜色标签区分：

```tsx
const getSeverityBadge = (severity: string) => {
  switch (severity) {
    case 'high':   return <span className="status-error">高危</span>;
    case 'medium': return <span className="status-stopped">中危</span>;
    default:       return <span className="badge badge-blue">低危</span>;
  }
};
```

当没有任何拦截记录时，页面显示"所有请求均通过安全检查"的空状态提示——这其实是一种正面反馈，让管理员放心。

## 八、七大页面横向对比

最后用一张表横向对比各页面的技术特征：

| 维度 | Dashboard | Agents | Workflows | Tools | Memory | Audit | Security |
|------|-----------|--------|-----------|-------|--------|-------|----------|
| 请求方式 | 单次 fetch | Promise.all | 单次 fetch | 单次 fetch | Promise.all + 按需加载 | Promise.all + 按需筛选 | 单次 fetch |
| 数据可视化 | Recharts 甜甜圈 | 模板网格 | ReactFlow 画布 | 树形工具列表 | 左右分栏 | 时间线 | 策略卡片 |
| 交互操作 | 无（纯展示） | 创建/启停/删除 | 拖拽/连线/保存 | 刷新 | 点击查看详情 | 过滤筛选 | 刷新 |
| 空状态处理 | 图表区域替换 | 模态框引导 | 模板引导创建 | 全局空状态 | 左右双空状态 | 时间线替换 | 正面文案 |
| 动画效果 | stagger 卡片淡入 | 模态框弹入 | 无 | 无 | 无 | 无 | 无 |
| 轮询/实时 | 否 | 否 | 否 | 否 | 否 | 否 | 否 |

你会发现七个页面都没有使用 WebSocket 或轮询——整个前端是"快照式"的，每次进入页面拉取最新数据。这是有意为之：Agent Studio 是一个管理面板，不是实时监控大屏，按需刷新的交互模型完全够用。

## 九、系列回顾与本阶段总结

写到这里，Agent Studio 系列五篇文章就全部完结了。让我简单回顾一下每篇的核心内容：

**第一篇**我们搭好了项目骨架——FastAPI + React + SQLite 的技术栈选型，前后端分离的开发模式，以及 Tailwind CSS 的配置方案。

**第二篇**深入后端 BFF 层，拆解了 FastAPI 路由设计、SQLite 本地存储、代理透传模式和 Dashboard 聚合的实现。

**第三篇**聚焦前端组件体系，从 StatCard、Badge、空状态到全局动画，建立了一套一致的 UI 语言。

**第四篇**是最硬核的部分——ReactFlow 工作流可视化编辑器。我们实现了 8 种自定义节点的拖拽连线、3 套预置工作流模板、以及 JSON 导入导出功能。

**第五篇**也就是本篇，从使用者的视角走遍了七个管理页面，看到了数据如何从后端流向前端，最终以卡片、图表、时间线的形式呈现。

如果要用一句话总结这个项目：**Agent Studio 是一个"够用就好"的 Agent 开发管理平台——它不追求 LangChain 那样的大而全，而是用最少的代码覆盖 Agent 从创建到审计的完整生命周期。**

整个项目大约 3000 行代码（后端 Python ~1100 行 + 前端 React/TS ~1900 行），部署只需要一个 SQLite 文件和一个 MCP Server 连接。对于个人开发者或小团队来说，这个体量足以支撑日常的 Agent 开发和调试需求。
